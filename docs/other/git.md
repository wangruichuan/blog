git中文件的状态：
1. 未跟踪：使用git add 添加文件到暂存区
2. 暂存：等待提交。表示文件的修改已被git记录，但还没保存到版本库中。
3. 已修改: 修改后未暂存
4. 已提交：`git commit -m "提交信息"`
   
- 从 暂存 回到 已修改 状态：`git restore --staged`
- 丢弃 已修改 的修改，恢复到最近提交的版本：`git restore`
- 暂存咱们的代码：常见的场景就是我现在在一个feature上开发，然后需要紧急切换到另一个分支改bug，但是当前feature分支还没有修改完，如果直接切换会提示切换失败，提示你要commit一下，但是这会导致你多一个黑历史，这时候就可以使用`git stash` 来暂存

## 变基
我们通过 merge 合并分支时，在提交记录中会将所有的分支创建和分支合并的过程全部都显示出来，这样当项目比较复杂，开发过程比较波折时，我必须要反复的创建、合并、删除分支。这样一来将会使得我们代码的提交记录变得极为混乱。

在开发中除了通过 merge 来合并分支外，还可以通过变基来完成分支的合并。

原理（变基时发生了什么）：

1. 当我们发起变基时，git 会首先找到两条分支的最近的共同祖先
2. 对比当前分支相对于祖先的历史提交，并且将它们提取出来存储到一个临时文件中
3. 将当前部分指向目标的基底
4. 以当前基底开始，重新执行历史操作

变基和 merge 对于合并分支来说最终的结果是一样的！但是变基会使得代码的提交记录更整洁更清晰！

注意！大部分情况下合并和变基是可以互换的，但是如果分支已经提交给了远程仓库，那么这时尽量不要变基。


## Git子模块

Git 子模块用于在一个 Git 仓库里嵌套另一个独立的 Git 仓库，标蓝且带哈希字符串（如 f809f35、d0dc976 ，这是子模块仓库对应提交的 commit ID ）

- 蓝色 + 哈希：是 Git 识别子模块的可视化标记，表明这些文件夹并非普通目录，而是关联了外部独立仓库的子模块。

- 随机字符串（commit ID）：记录子模块当前在父仓库中引用的具体版本，确保父仓库检出时，子模块能准确同步到对应状态。

通过子模块，可将独立维护的代码（如工具库、公共模块）“嵌入” 主项目，既分离仓库管理，又能在主项目里统一版本控制，方便多人协作和复杂项目拆分 。

若要操作子模块（更新、初始化等），需用 git submodule 系列命令（如 git submodule update --init ）。

### 常见步骤

![](https://pic1.imgdb.cn/item/68a2ac4e58cb8da5c82ce777.png)

1. 从主分支开一个新分支feature : `git checkout xxx`
2. 在这个新分支上开发
3. 提交PR，合并到主分支

- 如何命名新的分支？
  一般都有开发规范，比如：
  ①功能分支：feature/xxx 主要是用于开发新的特性
  ②发布分支：release/xxx 创建新的发布版本
  ③bug修复分支：fix/xxx 创建bug修复分支
  当然，还有其他的，比如develop分支，master分支....

- 如何创建PR?
  一般要填写如下内容：
  **Title**: 描述
  **Message**: 详细描述，如果Title已经足够，这里可以空白。

#### 代码开发流程

三个环境，对应三个分支：
1. 开发环境 dev ：冒烟测试-> 发送提测邮件
2. 测试环境 test ：测试收到邮件 -> 进入测试环境 -> 发送测试完成的邮件，最终会发送一个 上线邮件，抄送各大领导人
3. 生产环境 master ：一般都是晚上十点多

平时写代码在feature-1.0.0 分支，会不断的迭代版本号，功能开发完成后，就合并到dev分支，

**bug处理**

- T0:两小时之内修复的，全部门通报+扣钱
- 出现bug后，会拉一个hotfix分支出来去紧急修复代码，然后合并到测试环境，让测试去测，然后赶紧发布到生产，
