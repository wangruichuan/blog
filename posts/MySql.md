# 一、基本概念

1. 不熟悉的列类型：

   - decimal：特点：固定精度和小数位数  明确地指定总的数字位数以及小数部分的位数 例如`DECIMAL(5,2)`表示总的数字位数为 5，其中小数部分有 2 位  
     例如，如果定义一个字段为`DECIMAL(8,3)`，那么可以存储的值如 123.456 、9876.123 等，但不能存储 1234.567 （超出了总的位数）或 12.34 （小数位数不足）。
   - datetime VS timestamp：
     - `TIMESTAMP` 会跟随设置的时区变化而变化，而 `DATETIME` 保存的是绝对值，不会受到时区的影响。`TIMESTAMP` 储存占用 4 个字节，`DATETIME` 储存占用 8 个字节。
     - `TIMESTAMP` 可表示的时间范围为 1970-01-01 00:00:00 到 2038-01-09 03:14:07，`DATETIME` 支持的时间范围更宽，为 1000-01-01 00:00:00 到 9999-12-31 23:59:59。
     - `TIMESTAMP` 相对 `DATETIME` 索引速度更快，因为它更轻量。

2. 字段属性：

   - **UnSigned**：

   - **zerofill**：当使用`ZEROFILL`属性时，列的数据类型通常应为整数类型，如`INT`、`SMALLINT`、`BIGINT`等。`ZEROFILL`属性的作用是在显示数值时，用零填充左侧以达到指定的宽度。

     ```sql
     CREATE TABLE example_table (
         id INT(5) ZEROFILL,
         name VARCHAR(50)
     );
     ```

     列的显示宽度为`5`，使用`ZEROFILL`属性进行零填充。这意味着如果插入一个值为`1`的`id`，在检索数据时会以`00001`的形式显示。需要注意的是，`ZEROFILL`只是影响数据的展示，存储的数据仍然是实际的整数值

   - **自增、非空、默认**

   - 外键：删除有外键关系的表的时候，必须要先删除从表，再删除主表。

   - 每个表必须要有的字段：id，乐观锁，伪删除，创建、修改时间

   - 乐观锁：

     假设图书馆有一本书，很多人都可能想要借它。就好像多个事务都想要操作同一条数据。

     一个人（事务 A）先看到这本书在书架上（读取数据），他觉得自己能顺利借到（乐观地认为不会有冲突）。但在他去办理借阅手续的过程中，另一个人（事务 B）可能先一步把书借走了（数据被修改）。当事务 A 去办理借阅时，工作人员发现书已经被借走了（检查发现数据不一致），这时事务 A 就需要重新去查看还有哪些书可以借（重新读取数据再尝试操作）。

     在图书馆借还书的例子中，图书馆的借阅系统充当了乐观锁的角色。

     当第一个人看到书在书架上，这时候借阅系统中该书的状态是“可借”，这就相当于事务读取到了数据的初始状态。这个人认为自己能顺利借到，这就是乐观的态度。

     而在他去办理手续的过程中，第二个人来借书并完成了借阅流程，此时借阅系统会立刻将该书的状态更新为“已借”，这就像是数据被其他事务修改了。

     当第一个人去办理借阅时，借阅系统会检查该书的当前状态。如果仍然是“可借”，那就可以顺利办理借阅，相当于乐观锁检查通过，可以进行更新操作；但如果此时状态是“已借”，那就相当于乐观锁检查失败，发现了数据不一致的情况。这时，借阅系统就会反馈给这个人书已被借走的信息，让他重新去选择其他书或者采取其他行动，就如同在乐观锁机制中需要重新进行操作或采取其他策略一样。工作人员在这里也会根据借阅系统的状态提示来执行相应的流程，起到辅助和监督乐观锁机制运行的作用。

   **数据行锁定**：数据行锁定指的是在数据库操作中，对特定的一行数据进行锁定。
   当多个事务同时试图对同一行数据进行操作时，通过行锁定机制，可以确保在一个事务对该行数据进行操作（如修改、删除等）期间，其他事务不能同时对这同一行数据进行冲突的操作。这样可以避免数据的不一致性和冲突。
   比如说，事务 A 正在修改某一行数据，此时这一行就被锁定了，其他事务 B 想要同时修改这同一行数据就会被阻塞，直到事务 A 完成操作并释放锁定后，事务 B 才能继续对这一行进行操作。行锁定是数据库并发控制的一种重要手段，有助于维持数据的正确性和完整性，同时在一定程度上提高数据库操作的并发性和效率。

   > 那如果有乐观锁了，是否还需要数据行锁定？

   乐观锁主要适用于并发冲突不频繁的场景，它通过版本等方式来检测和处理冲突。但在一些对数据一致性要求非常高、并发冲突可能性较大或者操作必须严格串行执行的场景，仅靠乐观锁可能不够。

   数据行锁定可以更直接、绝对地保证特定行数据在某一时刻只能被一个事务操作，能避免一些复杂的冲突处理情况。

   例如，在一些关键业务流程中，可能需要更强有力的锁定机制来确保不会出现任何意外的数据不一致情况，这时行锁定就可能是必要的补充。所以两者可以根据具体的业务需求和场景特点来结合使用或单独使用，以达到最佳的效果。

4. **truncated**：在数据库中，“TRUNCATE”用于快速删除表中的所有数据并重置表的相关参数，比如自增索引等，它比逐行删除数据的效率要高很多。但需要注意的是，TRUNCATE 操作**无法回滚**，执行后数据将不可恢复。

# 四、索引

通过索引可以帮助我们更快的获取SQL执行的结果

常见的索引类型有：主键索引、唯一索引、常规索引、全文索引（仅限MyISAM）

AI对索引的解释：

> 索引可以想象成现实中一本书的目录。
>
> 在书中，目录列出了各个章节的标题以及对应的页码。当你想快速找到特定内容时，通过目录可以迅速定位到大概的位置，而不用逐页去翻找。
>
> 在数据库中也是类似的，索引就像是为数据建立的一个快速查找的“目录”。比如有一张很大的人员信息表，包含姓名、年龄等字段，如果建立了姓名的索引，那么当你要查找特定姓名的人员时，数据库就可以通过这个索引快速定位到相关的记录，而不必在所有数据中慢慢搜索，大大提高了查询的效率。就像你通过书的目录能快速找到想要的章节一样。
>
> **主键索引**：就如同这本书的唯一编号，它是独一无二且不能为空的。比如每本书都有一个唯一的 ISBN 码，这就类似于主键索引。
>
> **唯一索引**：可以想象成书中每个人的身份证号，是唯一的，但可能存在某些特殊情况导致部分位置为空（比如还没登记身份证的人）。
>
> **常规索引**：这就像根据书里的一些常见关键词做的索引，比如“爱情”“冒险”等，不要求它们是唯一的，只是为了方便快速找到相关主题的内容。
>
> **全文索引**：想象一下有一个很大的图书馆，里面有非常多的书籍。现在要在这个图书馆里找包含特定词语的所有书籍。如果没有全文索引，就相当于要一本本去翻阅所有的书来查找，这会非常耗时费力。
>
> 而有了全文索引，就好像图书馆有一个专门的系统，对每本书的内容都进行了分析和记录，当你输入特定词语时，它能快速地告诉你哪些书里包含了这个词语，就像搜索引擎一样。比如你想找所有提到“勇气”这个词的书，全文索引就能快速帮你筛选出来，而不必你自己去逐本翻找，极大地提高了查找的效率和便捷性。这样是不是能更好地理解全文索引啦？

虽然索引可以极大地提高查询效率，但索引也不是越多越好。

一方面，创建过多的索引会占用大量的存储空间，因为每个索引都需要额外的数据结构来存储。

另一方面，当对数据进行插入、更新和删除操作时，过多的索引需要同时进行维护，这会增加这些操作的开销，导致数据操作的性能下降。

而且，在某些情况下，<u>过多的索引可能会导致查询优化器选择了不恰当的索引</u>，反而降低了查询性能。此外，索引的维护也需要一定的系统资源和时间成本。所以，需要根据实际的业务需求和数据特点，合理地选择和创建索引，而不是一味地追求数量多。
