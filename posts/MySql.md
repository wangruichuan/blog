# 一、基本概念

1. 字段属性：

   - **zerofill**：当使用`ZEROFILL`属性时，列的数据类型通常应为整数类型，如`INT`、`SMALLINT`、`BIGINT`等。`ZEROFILL`属性的作用是在显示数值时，用零填充左侧以达到指定的宽度。

     ```sql
     CREATE TABLE example_table (
         id INT(5) ZEROFILL,
         name VARCHAR(50)
     );
     ```

     列的显示宽度为`5`，使用`ZEROFILL`属性进行零填充。这意味着如果插入一个值为`1`的`id`，在检索数据时会以`00001`的形式显示。需要注意的是，`ZEROFILL`只是影响数据的展示，存储的数据仍然是实际的整数值

   - **自增、非空、默认**

   - 外键：删除有外键关系的表的时候，必须要先删除从表，再删除主表。

   - 每个表必须要有的字段：id，乐观锁，伪删除，创建、修改时间

   - 乐观锁：

     假设图书馆有一本书，很多人都可能想要借它。就好像多个事务都想要操作同一条数据。

     一个人（事务 A）先看到这本书在书架上（读取数据），他觉得自己能顺利借到（乐观地认为不会有冲突）。但在他去办理借阅手续的过程中，另一个人（事务 B）可能先一步把书借走了（数据被修改）。当事务 A 去办理借阅时，工作人员发现书已经被借走了（检查发现数据不一致），这时事务 A 就需要重新去查看还有哪些书可以借（重新读取数据再尝试操作）。

     在图书馆借还书的例子中，图书馆的借阅系统充当了乐观锁的角色。

     当第一个人看到书在书架上，这时候借阅系统中该书的状态是“可借”，这就相当于事务读取到了数据的初始状态。这个人认为自己能顺利借到，这就是乐观的态度。

     而在他去办理手续的过程中，第二个人来借书并完成了借阅流程，此时借阅系统会立刻将该书的状态更新为“已借”，这就像是数据被其他事务修改了。

     当第一个人去办理借阅时，借阅系统会检查该书的当前状态。如果仍然是“可借”，那就可以顺利办理借阅，相当于乐观锁检查通过，可以进行更新操作；但如果此时状态是“已借”，那就相当于乐观锁检查失败，发现了数据不一致的情况。这时，借阅系统就会反馈给这个人书已被借走的信息，让他重新去选择其他书或者采取其他行动，就如同在乐观锁机制中需要重新进行操作或采取其他策略一样。工作人员在这里也会根据借阅系统的状态提示来执行相应的流程，起到辅助和监督乐观锁机制运行的作用。

   **数据行锁定**：数据行锁定指的是在数据库操作中，对特定的一行数据进行锁定。
   当多个事务同时试图对同一行数据进行操作时，通过行锁定机制，可以确保在一个事务对该行数据进行操作（如修改、删除等）期间，其他事务不能同时对这同一行数据进行冲突的操作。这样可以避免数据的不一致性和冲突。
   比如说，事务 A 正在修改某一行数据，此时这一行就被锁定了，其他事务 B 想要同时修改这同一行数据就会被阻塞，直到事务 A 完成操作并释放锁定后，事务 B 才能继续对这一行进行操作。行锁定是数据库并发控制的一种重要手段，有助于维持数据的正确性和完整性，同时在一定程度上提高数据库操作的并发性和效率。

   > 那如果有乐观锁了，是否还需要数据行锁定？

   乐观锁主要适用于并发冲突不频繁的场景，它通过版本等方式来检测和处理冲突。但在一些对数据一致性要求非常高、并发冲突可能性较大或者操作必须严格串行执行的场景，仅靠乐观锁可能不够。

   数据行锁定可以更直接、绝对地保证特定行数据在某一时刻只能被一个事务操作，能避免一些复杂的冲突处理情况。

   例如，在一些关键业务流程中，可能需要更强有力的锁定机制来确保不会出现任何意外的数据不一致情况，这时行锁定就可能是必要的补充。所以两者可以根据具体的业务需求和场景特点来结合使用或单独使用，以达到最佳的效果。

3. **truncated**：在数据库中，“TRUNCATE”用于快速删除表中的所有数据并重置表的相关参数，比如自增索引等，它比逐行删除数据的效率要高很多。但需要注意的是，TRUNCATE 操作**无法回滚**，执行后数据将不可恢复。

# 三、事务

>  默认MySQL的事务是自动提交的，也就是说，当执行完一条DML语句时，MySQL会立即隐式的提交事务。
>
> 除了在那种已经禁止了自动提交（AUTOCOMMIT）的事务场景下可以使用“LOCKTABLES”（锁表操作），在其他的任何情形下，不管当前使用的是哪种具体的存储引擎，都不应该明确地去执行“LOCKTABLES”这个操作。

## 3.1 ACID

原子：要么都执行，要么都不执行

一致：事务之前前后，状态是一致的

持久：一旦提交，事务所做的修改就会被永久保存到数据库中。

隔离：一个事务所做的修改在最终提交以前，对其他事务是不可见的

## 3.2 隔离级别

1. 未提交读：**在事务中可以查看其他事务中还没有提交的修改**。这个隔离级别会导致很多问题，从性能上来说，不会比其他级别好太多，却缺乏其他级别的很多好处，除非有非常必要的理由，在实际应用中一般很少使用。

   读取未提交的数据，也称为脏读(dirty read)。

2. 提交读：**一个事务可以看到其他事务在它开始之后提交的修改，但在该事务提交之前，其所做的任何修改对其他事务都是不可见的。**然而，这种级别仍然存在“不可重复读”的情况。这意味着在同一个事务中，如果先后两次执行相同的语句，由于其他事务可能在这期间提交了新的修改，所以该事务第二次执行时可能会看到与第一次不同的数据结果。

3. 可重复读：**一个事务在执行过程中多次读取同样的数据时，无论其他事务是否进行了修改提交，该事务始终能得到相同的结果，即不会看到其他事务对数据的修改，避免了不可重复读的现象。**
   例如，事务 T1 开启后，在可重复读级别下，它读取了某些数据。在 T1 执行期间，即使其他事务对这些数据进行了修改并提交，T1 后续再次读取这些数据时，仍然看到的是它第一次读取时的数据状态。这有助于确保事务在执行过程中数据的稳定性和一致性，不会因为其他事务的干扰而出现前后读取数据不一致的情况。但需要注意的是，可重复读级别下仍然可能存在幻读的问题。

   > 具体来说，幻读是指在一个事务中，按照某个条件进行查询，第一次查询时没有发现符合条件的行，但是在随后的操作中（比如插入新行等），却又发现了符合该条件的行，就好像出现了“幻觉”一样。
   >
   > 比如，事务 A 按照某个特定条件查询，没有找到相关记录。此时另一个事务 B 插入了一些满足该条件的新记录并提交。然后事务 A 再次按照相同条件查询时，却发现了之前不存在的这些记录，这就是幻读现象。幻读主要涉及到对记录数量或范围的“错觉”，与不可重复读主要针对特定数据值的不一致有所不同。这种情况可能会对一些基于查询结果进行后续操作的事务产生影响，导致结果不符合预期。

4. 串行化：没啥可说的，一个事务执行时，不允许别的事务执行，基本上没有用这种隔离级别的

## 3.3 事务日志

**事务日志有助于提高事务的效率。**

> 事务日志就像是一个记录员。
>
> 把它想象成一个专门记录重要事情的本子。当数据库要进行一些操作，比如修改数据的时候，它不会马上就直接在真正的数据文件上改动，而是先在这个“记录员”本子（事务日志）上写下自己要做什么，比如“我要把这个数据改成什么什么样子”。这样做的好处是速度快，不用直接费力地去改数据文件。然后呢，这些记录会被好好地保存到硬盘上。以后呢，会有个专门的“工作人员”（后台进程）按照这个记录去真正地修改数据文件。
>
> 而且就算在这个过程中，突然出了什么问题，比如电脑崩溃了，等重新启动后，通过这个事务日志，还能知道之前要做什么，然后把没做完的事情接着做完，就像根据记录恢复之前的情况一样。

​		存储引擎只需要更改内存中的数据副本，而不用每次修改磁盘中的表，这会非常快。然后再把更改的记录写入事务日志中，事务日志会被持久化保存在硬盘上。因为事务日志采用的是追加写操作，是在硬盘中一小块区域内的顺序I/0，而不是需要写多个地方的随机I/0，所以写入事务日志是一种相对较快的操作。最后会有一个后台进程在某个时间去更新硬盘中的表。

​		如果修改操作已经写入事务日志，那么即使系统在数据本身写入硬盘之前发生崩溃，存储引擎仍可在重新启动时恢复更改。具体的恢复方法则因存储引擎而异。

**事务日志通常会同时存在于内存和磁盘中。**

​		在数据库系统中，事务日志主要用于记录数据库的修改操作，以确保数据的一致性和可恢复性。当事务提交时，相关的修改操作会首先被写入内存中的日志缓冲区，然后再异步地刷新到磁盘上的事务日志文件中。

> 那如果事务日志还没来得及存在磁盘上，只存到了内存上，这时候电脑崩溃怎么办

​		这种情况下是有可能无法完全恢复的。

​		如果事务日志没有及时写入磁盘，当电脑崩溃后，内存中的数据会丢失，那么那些只存在于内存中的还未持久化的事务日志信息也就丢失了，就无法依据这些来完整地恢复所有未完成的事务，可能会导致部分数据的不一致或丢失。但一些数据库系统可能会采取一些其他的技术或策略来尽量降低这种风险和影响，比如设置更频繁的内存刷新到磁盘的机制等。不过，总体来说，这种情况确实会给数据恢复带来不确定性和潜在问题。

## 3.4 多版本并发控制

用来处理并发事务的机制。

MVCC的工作原理是使用数据在某个时间点的快照来实现的。这意味着，无论事务运行多长时间，都可以看到数据的一致视图，也意味着不同的事务可以在同一时间看到同一张表中的不同数据!如果你之前没有这方面的概念,这句话听起来可能有点让人迷惑，熟悉了以后你会发现还是很容易理解的。

**主要特点**：

- 允许不同的事务读取到不同版本的数据，从而在一定程度上避免了读-写和写-读冲突。
- 实现了非阻塞的读操作，即读操作不会因为有其他事务正在执行写操作而被阻塞。

**原理**：

- 每行数据会有多个版本，每个版本都记录了创建它的事务 ID。
- 事务在执行读取操作时，根据事务的隔离级别等规则，找到符合条件的版本来读取。
- 当事务进行写入时，会创建一个新的数据版本，并标记相关信息。

例如，事务 A 读取了某行数据，之后事务 B 对该行进行了修改并提交。此时如果事务 A 再次读取该行，它可能仍然能看到自己之前读取的版本，而不会被事务 B 的修改所影响，除非事务 A 本身也进行了提交或有其他符合特定条件的操作。

MVCC 使得 MySQL 在并发场景下能够提供较好的性能和并发性，减少了因为锁竞争导致的阻塞和性能下降问题。

# 四、索引

通过索引可以帮助我们更快的获取SQL执行的结果 。常见的索引类型有：主键索引、唯一索引、常规索引、全文索引（仅限MyISAM）

> 索引可以想象成现实中一本书的目录。
>
> 在书中，目录列出了各个章节的标题以及对应的页码。当你想快速找到特定内容时，通过目录可以迅速定位到大概的位置，而不用逐页去翻找。
>
> 在数据库中也是类似的，索引就像是为数据建立的一个快速查找的“目录”。比如有一张很大的人员信息表，包含姓名、年龄等字段，如果建立了姓名的索引，那么当你要查找特定姓名的人员时，数据库就可以通过这个索引快速定位到相关的记录，而不必在所有数据中慢慢搜索，大大提高了查询的效率。就像你通过书的目录能快速找到想要的章节一样。
>
> **主键索引**：就如同这本书的唯一编号，它是独一无二且不能为空的。比如每本书都有一个唯一的 ISBN 码，这就类似于主键索引。
>
> **唯一索引**：可以想象成书中每个人的身份证号，是唯一的，但可能存在某些特殊情况导致部分位置为空（比如还没登记身份证的人）。
>
> **常规索引**：这就像根据书里的一些常见关键词做的索引，比如“爱情”“冒险”等，不要求它们是唯一的，只是为了方便快速找到相关主题的内容。
>
> **全文索引**：想象一下有一个很大的图书馆，里面有非常多的书籍。现在要在这个图书馆里找包含特定词语的所有书籍。如果没有全文索引，就相当于要一本本去翻阅所有的书来查找，这会非常耗时费力。
>
> 而有了全文索引，就好像图书馆有一个专门的系统，对每本书的内容都进行了分析和记录，当你输入特定词语时，它能快速地告诉你哪些书里包含了这个词语，就像搜索引擎一样。比如你想找所有提到“勇气”这个词的书，全文索引就能快速帮你筛选出来，而不必你自己去逐本翻找，极大地提高了查找的效率和便捷性。这样是不是能更好地理解全文索引啦？

虽然索引可以极大地提高查询效率，但索引也不是越多越好。

一方面，创建过多的索引会占用大量的存储空间，因为每个索引都需要额外的数据结构来存储。

另一方面，当对数据进行插入、更新和删除操作时，过多的索引需要同时进行维护，这会增加这些操作的开销，导致数据操作的性能下降。

而且，在某些情况下，<u>过多的索引可能会导致查询优化器选择了不恰当的索引</u>，反而降低了查询性能。此外，索引的维护也需要一定的系统资源和时间成本。所以，需要根据实际的业务需求和数据特点，合理地选择和创建索引，而不是一味地追求数量多。

## 4.1 索引基础

MySQL中索引是如何工作的,最简单的方法就是去看看一本书的“索引”部分:如果想在一本书中找到某个特定主题，一般会先看书的“索引”，找到对应的页码。

索引可以包含一个或多个列的值。如果索引包含多个列，那么列的顺序也十分重要，因为**MySQL只能高效地使用索引的最左前缀列**。<u>创建一个包含两个列的索引，和创建两个只包含一列的索引是大不相同的</u>。

> 如果使用的是 ORM，是否还需要关心索引?
>
> 仍然需要理解索引，即使是使用对象关系映射(ORM)工具
>
> ORM 工具能够生产符合逻辑的、合法的查询(多数时候)，除非只是生成非常基本的查询(例如仅是根据主键查询)，否则它很难生成适合索引的查询。
>
> 无论是多么复杂的 ORM 工具，在精妙和复杂的索引面前都是“浮云”。

   B-Tree对索引列是顺序组织存储的，所以很适合查找范围数据。

   > 比如，在进行一个连续区间的数据查询时，利用其顺序结构可以很方便地依次找到相关的数据，而无需进行多次跳跃式的查找。
   >
   > 当进行连续区间的数据查询时，首先通过 B+树的树形结构快速定位到起始值所在的叶子节点位置，这一步利用了树形结构高效的检索能力。然后，由于叶子节点之间的顺序连接，就可以顺着这个顺序从起始值所在的节点开始，依次向后查找，轻松地找到区间内其他的顺序值，而不需要再进行复杂的跳跃或多次查找操作。这种方式极大地提高了范围查询的效率和便捷性，充分体现了 B+树在处理这类查询时的优势。
   >
   > 打个比方，就好像在一本书中查找某个段落，先通过目录快速找到起始页，然后顺着页面的顺序就能很方便地找到该段落中的其他内容。

   ------

   > 有一个名为People的表，其中包含last_name（姓）、first_name（名）和dob（出生日期）三个列，并在这三个列上创建了一个索引

   这里说的是在这三个列上创建了一个索引，意味着是一个组合索引，将这三个列的信息作为一个整体来构建索引。而不是对应三个索引。比如，如果有查询需要同时根据姓、名和出生日期的特定组合来查找记录，那么这个组合索引就能发挥作用，而不是分别为每个列单独创建一个索引。当然，在实际应用中，也可能根据具体需求同时存在单独索引和组合索引。

   **可以使用 B-Tree 索引的查询类型**（哪些情况下会使用到索引进行查询）:

   1. 全值匹配：全值匹配指的是和索引中的所有列进行匹配，例如前面提到的索引可用于查找姓名为 CubaAllen、出生于 1960-01-01的人。
   2. 匹配最左前缀：前面提到的索引可用于查找所有姓为A1len的人，即只使用索引的第一列。
   3. 匹配列前缀：也可以只匹配某一列的值的开头部分。例如前面提到的索引可用于查找所有以J开头的姓的人。这里也只使用了索引的第一列。
   4. 匹配范围值：例如前面提到的索引可用于查找姓在Allen和Barrymore之间的人。这里也只使用了索引的第一列。
   5. 精确匹配某一列并范围匹配另外一列：前面提到的索引也可用于查找所有姓为A1len，并且名字是字母K开头(比如Kim、Karl 等)的人。即第一列 last name 全匹配，第二列 first_name 范围匹配。
   6. 只访问索引的查询：在某些情况下，进行查询操作时，通过索引本身就能够完全满足查询需求，获取到所需的全部信息，而不需要再进一步去访问实际的数据行。比如说，假设索引中包含了某些关键信息，通过对该索引的检索就能够确定符合条件的结果，无需再去查看具体每一行数据中的其他详细内容。

   **不能使用 B-Tree 索引的情况**:

   1. 如果不是按照索引的最左列开始查找，则无法使用索引。例如上面例子中的索引无法用于查找名字为Bi11的人，也无法查找某个特定生日的人，因为这两列都不是最左数据列。类似地，也无法查找姓氏以某个字母结尾的人。
   2. 不能跳过索引中的列。也就是说，前面所述的索引无法用于查找姓为Smith并且在某个特定日期出生的人。如果不指定名(first name)，则MySQL只能使用索引的第一列。
   3. 如果查询中有某个列的范围查询，则其右边所有列都无法使用索引优化查找。例如有查询 `WHERE last name='Smith'AND first name LIKE'J%'AND dob ='1976-12-23'`，这个查询只能使用索引的前两列，因为这里LIKE是一个范围条件。如果范围查询列值的数量有限，那么可以通过使用多个等于条件来代替范围条件。

2. **哈希索引**
   哈希索引基于哈希表实现，只有**精确匹配索引所有列**的査询才有效。

    对于每一行数据，存储引擎都会对所有的索引列计算一个哈希码，哈希码是一个较小的值，并且不同键值的行计算出来的哈希码也不一样。哈希索引将所有的哈希码存储在索引中，同时在哈希表中保存指向每个数据行的指针，在 MySQL中，只有Memory引擎显式支持哈希索引。 		

   InnoDB引擎有一个特殊的功能叫做“**自适应哈希索引**”。当InnoDB注意到某些索引值被使用得非常频繁时，它会在内存中基于B-Tree索引之上再创建一个哈希索引，这样就让 B-Tree索引也具有哈希索引的一些优点，比如快速的哈希查找。这是一个完全自动的、内部的行为，用户无法控制或者配置，不过如果有必要完全可以关闭该功能。 

   例如，假设有一个用户表，其中包含了用户的 ID、姓名和年龄等列。如果经常需要根据用户 ID 来查询用户信息，那么 InnoDB 引擎可能会自动为用户 ID 列创建一个自适应哈希索引。这样，当再次根据用户 ID 进行查询时，就可以直接使用哈希索引来快速定位到对应的用户记录，而无需进行全表扫描或 B 树索引的查找，从而提高查询性能。

3. **全文索引**

   全文索引是一种特殊类型的索引，它查找的是文本中的关键词，而不是直接比较索引中的值。全文搜索和其他几类索引的匹配方式完全不一样。它有许多需要注意的细节，如停用词、词千和复数、布尔搜索等。全文索引更类似于搜索引擎做的事情，而不是简单的 WHERE 条件匹配。

   全文索引和一般的索引不一样，一般索引就是比对着某个具体值来找，比如找数字 5 啊之类的。但全文索引，它是在一大段文字里找特定的词语。

   全文搜索的方式和其他索引找的办法完全不同。用它的时候有好多要注意的地方，像有些词没啥意义，比如“的”“啊”这些就可能被忽略掉，还有词的变化呀，比如复数形式、词干啥的，还有类似搜索引擎那样的布尔搜索，可以用一些逻辑关系去搜。

   总之呢，全文索引不是简单地像平常条件判断那样找，它更像是搜索引擎干的事儿，能在大量文字里找你想要的那些关键词。

### 4.2.2 前缀索引和索引选择性

有时候需要索引很长的字符列，这会让索引变得大且慢。一个策略是前面提到过的模拟哈希索引。但有时候这样做还不够，还可以做些什么呢?

通常可以索引开始的部分字符，这样可以大大节约索引空间，从而提高索引效率。但这样也会降低索引的选择性。

索引的选择性是指，不重复的索引值和数据表的记录总数的比值。**索引的选择性越高则查询效率越高**，因为选择性高的索引可以让MySOL在查找时过滤掉更多的行。唯一索引的选择性是1，这是最好的索引选择性，性能也是最好的。

对于 BL0B、TEXT或者很长的 VARCHAR类型的列，必须使用前缀索引，因为MySOL不允许索引这些列的完整长度。

诀窍在于要选择足够长的前缀以保证较高的选择性，同时又不能太长。前缀应该足够长，以使得前缀索引的选择性接近于索引整个列。

### 4.2.3 多列索引

很多人对多列索引的理解都不够。一个常见的错误就是，为每个列创建独立的索引，或者按照错误的顺序创建多列索引。

![image-20240601101114722](https://cdn.jsdelivr.net/gh/wangruichuan/images@main/2024/image-20240601101114722.png)

在多个列上建立独立的单列索引大部分情况下并不能提高MySQL的查询性能。这样一来最好的情况下也只能是“一星”索引，其性能比起真正最优的索引可能差几个数量级。有时如果无法设计一个“三星”索引，那么不如忽略掉 WHERE子句，集中精力优化索引列的顺序，或者创建一个全覆盖索引（索引包含了查询中需要的所有列的数据）。

虽然MySQL有一种”索引合并“的策略，但是尽量还是别这样搞。

### 4.2.4 合适的索引列顺序

**把最能区分不同数据的那一列放在索引的开头**。

这个建议有用吗?在某些场景可能有帮助，但通常不如避免随机I0和排序那么重要，考虑问题需要更全面。

当不涉及排序和分组操作，只是单纯为了优化 WHERE 条件的查找时，把选择性最高的列放在索引前面通常是个不错的做法。因为这样设计的索引可以最快地筛选出所需的行，尤其对于那些只使用索引部分前缀列的查询，其选择性会更高。然而，索引性能并非仅仅取决于所有索引列的选择性（整体基数），还和查询条件中的具体值，也就是值的分布情况有关。这就如同选择前缀长度时一样，可能需要根据实际运行中那些频率最高的查询来灵活调整索引列的顺序，以确保在这种情况下索引能达到最高的选择性，从而更好地提升性能。

![image-20240601104205604](https://cdn.jsdelivr.net/gh/wangruichuan/images@main/2024/image-20240601104205604.png)

是应该创建一个(staff id,customer id)索引还是应该颠倒一下顺序?需要进行测试。

例如，在某些应用程序中，未登录用户的用户名被记录为“guest”，在记录用户行为的会话表和其他记录用户活动的表中，“guest”就成为了一个特殊的用户 ID。一旦涉及到这个用户的查询，可能性能就会大不相同。

这里说尽管建立了索引但查询还是慢，是因为虽然建了索引，但可能没有充分考虑到像“guest”这类特殊值的高基数情况。也就是说，对于这类特殊值，它在数据中出现的频率非常高，可能导致索引不能很好地发挥作用，或者说索引在面对这种大量重复的特殊值时，其优化效果被大大削弱了，所以才会出现查询运行慢的情况呀。这也提醒我们在设计索引时，不能只看表面是否建立了索引，还要深入分析数据的具体特点和分布情况。

### 4.3.5 聚簇索引

在数据库中，InnoDB存储引擎用聚簇索引来组织数据，就像书的目录一样，它不仅记录了数据的“标题”（也就是索引），还记录了数据“内容”（也就是数据行）。

想象一下，如果在图书馆找书，首先会看目录找到书的标题，然后再找到书的具体位置。但在InnoDB的聚簇索引中，不需要再走两步，因为目录（索引）和书的内容（数据行）是放在一起的。

聚簇索引的好处是，它让查找数据变得更快，因为你可以一步到位，直接找到数据。但是，也有一个限制，就是每本书（表）只能有一个这样的目录（聚簇索引）。不过，你可以用一种叫做“覆盖索引”的方法来绕过这个限制，这就像是在目录里添加一些额外的信息，让你可以更快地找到你需要的内容。

那我们来看几个问题：

1. 就是每本书（表）只能有一个这样的目录（聚簇索引），这是为什么？

   1. **数据唯一性**：书架上的每本书都是唯一的，你不能把同一本书放在两个不同的位置。同样，数据库表中的数据行也只能存放在一个地方，这样才能保证数据的一致性和完整性。
   2. **效率问题**：如果书架上有多个标签，你可能会混淆，不知道该看哪个标签来找到书。同样，如果一个表有多个聚簇索引，数据的物理存储就会变得复杂，这会降低数据访问的效率。
   3. **简化设计**：只有一个标签（聚簇索引）可以让数据库的设计和维护变得更加简单。
   4. **性能考虑**：聚簇索引可以提高数据访问的速度，因为它让数据和索引紧密地结合在一起。如果有很多聚簇索引，这种优化就会受到影响，因为数据库需要在多个索引之间进行选择和维护。

2. 可以用一种叫做“覆盖索引”的方法来绕过这个限制，这个指的是什么，怎么理解？

   想象一下，你的书架上现在有一个特殊的标签，这个标签不仅告诉你书名，还提供了一些额外的信息，比如作者、出版日期、甚至是书的主要内容摘要。这样的标签可以帮你更快地决定是否这本书就是你要找的那本，而不需要你每次都去翻阅整本书。

   在数据库中，覆盖索引的作用与这个特殊的标签类似：

   1. **包含索引键**：覆盖索引首先包含了索引键，就像标签上写着书名。
   2. **包含数据行**：覆盖索引还包含了一些或全部的数据行，这就像是标签上还提供了作者、出版日期和内容摘要。
   3. **减少数据访问**：因为覆盖索引包含了额外的信息，所以有时候你不需要去访问表中的实际数据行，就可以得到你需要的信息。这就像是通过看标签上的额外信息，你就能决定是否需要阅读这本书。
   4. **提高查询效率**：使用覆盖索引可以减少对主数据表的访问次数，从而提高查询效率。这就像是通过看标签，你就能快速决定是否需要这本书，而不需要每次都去翻阅整本书。
   5. **模拟多个索引**：虽然一个表只能有一个聚簇索引，但是通过创建多个覆盖索引，你可以在某种程度上模拟多个聚簇索引的效果。每个覆盖索引都可以针对不同的查询需求提供优化。

   所以，覆盖索引就像是在目录里添加一些额外的信息，让你可以更快地找到你需要的内容，同时还能减少对原始数据的访问，提高查询的效率。

3. 覆盖索引是否是每本书都会有？

   - **不是每本书都有**：并不是书架上每本书的标签都包含额外的信息。只有那些你经常需要查找特定信息的书籍，你可能会为它们创建一个包含额外信息的标签。
   - **选择性创建**：在数据库中，你会选择性地为那些经常一起被查询的列创建覆盖索引。比如，如果你经常根据作者和书名来查找书籍，那么可以创建一个覆盖索引，包含作者和书名这两个字段。
   - **优化查询**：创建覆盖索引的目的是为了优化那些特定的查询操作，使得数据库能够更快地返回查询结果。

   所以，覆盖索引更像是你为那些特别重要或经常查询的书籍创建的特殊标签，而不是每本书都有。通过这种方式，你可以提高查找这些书籍的效率，而不需要每次都去翻阅整本书。

4. 数据行如果很大很多，会有什么问题？

   数据行的大小直接影响数据库的性能和存储效率。过大的数据行可能导致存储空间浪费、查询速度降低、索引效率下降、内存使用增加和I/O操作负担加重。为了优化性能，应考虑合理设计数据模型、使用索引、数据压缩、规范化、分页、选择合适的数据类型、定期清理数据以及合理配置数据库。

### 4.3.6 覆盖索引

假设有一个包含员工姓名、年龄和部门的表，经常执行这样的查询："查找所有30岁的员工的姓名"。在这种情况下，可以创建一个覆盖索引，只包含年龄和姓名这两个列。当执行上述查询时，数据库可以直接从索引中获取所有30岁员工的姓名，而不需要去访问数据表中的数据行。

这就是覆盖索引。叶子结点就已经包含了要查询的数据。

### 4.3.7 使用索引扫描来做排序

MySQL有两种方式可以生成有序的结果:

​	通过排序操作;

​	按索引顺序扫描;

如果EXPLAIN 出来的 type列的值为“index”，则说明MySQL使用了索引扫描来做排序

举个例子，如果你有一个查询如下：

```
SELECT column1, column2 FROM table1 ORDER BY column2;
```

并且`column2`上有索引，那么`EXPLAIN`的输出可能会显示`type`列的值为`"index"`，这意味着MySQL将使用`column2`上的索引来扫描数据并进行排序。总之，如果`EXPLAIN`的`type`列显示`"index"`，这通常是一个性能上的好消息，因为它意味着MySQL正在使用索引来优化查询的执行，包括排序操作。

通常情况下，扫描索引是很快的，因为这个过程只需要顺序地访问索引中的记录，从一个索引项移动到下一个索引项。

如果索引不能覆盖查询所需的全部列，也就是说，索引中不包含查询所要检索的所有字段，那么数据库在扫描索引后，还需要对每个索引项进行回表查询，也就是去数据表中查找对应的完整数据行。

回表查询通常涉及到随机I/O操作，因为数据表中的数据行可能不是顺序存储的，所以每次查找可能都需要在磁盘上随机定位到数据行的位置。

如果需要频繁进行回表查询，那么按索引顺序读取数据的速度可能会比顺序地进行全表扫描慢，因为全表扫描是顺序地读取数据，而不需要随机跳转。

在I/O密集型的工作负载下，即系统的性能瓶颈主要是磁盘I/O操作时，随机I/O操作（回表查询）会比顺序I/O操作（全表扫描）更加影响性能。

重复索引是指在相同的列上按照相同的顺序创建的相同类型的索引。

### 4.3.9 索引和锁

通过使用索引，查询可以更快地定位到所需的数据行，从而减少访问不需要的行，进而减少锁定的行数。尽管InnoDB的行锁效率较高且内存使用较少，但锁定行本身仍然会带来额外的开销。锁定超过需要的行会增加锁争用，减少并发性。通过减少锁定的行数，可以提高并发性能。InnoDB只在访问行时才会对其加锁。索引通过减少InnoDB需要访问的行数，间接减少锁的数量。索引只有在能够在存储引擎层过滤掉所有不需要的行时，才能有效减少锁定的行数。如果索引无法过滤掉无效的行，InnoDB检索到数据并返回给服务器层后，MySQL服务器才能应用`WHERE`子句进行过滤。这时，锁定行的操作已经发生，无法避免。

索引通过减少访问不需要的行数，有助于减少锁定的行数，从而提高数据库的性能，尤其是在并发环境下。InnoDB的锁机制和MySQL服务器层的过滤能力也在这一过程中发挥了作用。随着MySQL版本的更新，锁的处理机制也得到了改进，使得锁的释放更加及时和高效。

# 六、数据类型优化

## 6.1 原则

1. 更小的通常更好。
2. 简单就好：简单数据类型的操作通常需要更少的CPU周期。例如，整型比字符操作代价更低（因为字符集和校对规则(排序规则)使字符比较比整型比较更复杂）这里有两个例子：一个是应该使用 MySQL内建的类型，而不是字符串来存储日期和时间，另外一个是应该用整型存储IP地址。稍后我们将专门讨论这个话题。
3. 尽量避免 NULL：如果查询中包含可为NULL的列，对MySQL来说更难优化

> 例如，`DATETIME`和`TIMESAMP`都可以存储相同类型的数据:时间和日期，精确到秒。
>
> 然而 `TIMESTAMP` 只使用 `DATETIME` 一半的存储空间，并且会根据时区变化，具有特殊的自动更新能力。
>
> 另一方面，`TIMESTAMP`允许的时间范围要小得多，有时候它的特殊能力会成为障碍。
>
> MySOL为了兼容性支持很多别名，例如INTEGER、B00L以及 NUMERIC。它们都只是别名。

## 6.2 实数类型

- **decimal**：

特点：固定精度和小数位数  

明确地指定总的数字位数以及小数部分的位数 例如`DECIMAL(5,2)`表示总的数字位数为 5，其中小数部分有 2 位  
例如，如果定义一个字段为`DECIMAL(8,3)`，那么可以存储的值如 123.456 、9876.123 等，但不能存储 1234.567 （**超出了总的位数**）或 12.34 （**小数位数不足**）。

## 	6.4 Enum代替字符串类型

有时候可以使用枚举列代替常用的字符串类型。

枚举列可以把一些不重复的字符串存储成一个预定义的集合。

MySQL在存储枚举时非常紧凑，会根据列表值的数量压缩到一个或者两个字节中。

MySOL在内部会将每个值在列表中的位置保存为整数，并且在表的 .m 文件中保存“数字-字符串”映射关系的“查找表”。
